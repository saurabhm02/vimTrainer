# Database Schema: VimTrainer
**Version**: 1.0
**Last Updated**: 2026-06-16
**Author**: Backend Architect
**Status**: Production-Ready

---

## Overview

All tables use PostgreSQL 15 with UUID primary keys generated by `gen_random_uuid()`. Timestamps are `TIMESTAMPTZ` (stored UTC, displayed in user timezone by the frontend). Soft deletes use a `deleted_at` nullable column — queries must include `WHERE deleted_at IS NULL` to exclude soft-deleted rows. Enum-like constraints are enforced by `CHECK` constraints rather than PostgreSQL `ENUM` types to allow future value additions without a table lock.

Migration tool: `golang-migrate`. Files follow the naming convention `{NNN}_{description}.up.sql` / `{NNN}_{description}.down.sql`. Migrations run sequentially and are applied atomically per file. Use `-- migrate:no-transaction` only for `CREATE INDEX CONCURRENTLY` statements.

## Table Definitions

### 1. users

Registered user accounts. Guest sessions are tracked via `is_guest = TRUE` and a `guest_token` UUID. When a guest converts to a registered account, `is_guest` is set to `FALSE`, `guest_token` is nulled, and `email`/`password_hash` are populated. Soft delete: setting `deleted_at` disables login and hides all user data from queries.

```sql
CREATE TABLE users (
    id             UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    email          TEXT         UNIQUE,                       -- NULL for guest users
    password_hash  TEXT,                                      -- NULL for guest users
    is_guest       BOOLEAN      NOT NULL DEFAULT FALSE,
    guest_token    UUID         UNIQUE,                       -- NULL for registered users; set for guests
    display_name   TEXT         NOT NULL DEFAULT '',
    current_streak INT          NOT NULL DEFAULT 0 CHECK (current_streak >= 0),
    longest_streak INT          NOT NULL DEFAULT 0 CHECK (longest_streak >= 0),
    last_active_date DATE,                                    -- UTC date of last completed daily queue
    created_at     TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at     TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    deleted_at     TIMESTAMPTZ,                               -- soft delete; NULL = active

    -- Registered users must have email + password; guests must have guest_token
    CONSTRAINT chk_user_identity CHECK (
        (is_guest = FALSE AND email IS NOT NULL AND password_hash IS NOT NULL AND guest_token IS NULL)
        OR
        (is_guest = TRUE AND email IS NULL AND password_hash IS NULL AND guest_token IS NOT NULL)
    )
);

-- Active email uniqueness: enforced as partial unique index (excludes deleted and guest users)
CREATE UNIQUE INDEX idx_users_email_active
    ON users(email)
    WHERE deleted_at IS NULL AND is_guest = FALSE;

-- Guest token lookup: used by GuestAuth middleware on every practice request
CREATE UNIQUE INDEX idx_users_guest_token
    ON users(guest_token)
    WHERE guest_token IS NOT NULL;

-- Cleanup job: find expired soft-deleted accounts for hard delete
CREATE INDEX idx_users_deleted_at
    ON users(deleted_at)
    WHERE deleted_at IS NOT NULL;
```

### 2. keymap_sources

Tracks the origin of each import batch. One row per import operation. Keymaps reference their source for attribution in the "Imported Keymaps" UI. Soft delete allows the user to remove an import source from their library without hard-deleting the source record (used for audit trail and re-import prevention UI).

```sql
CREATE TABLE keymap_sources (
    id           UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    source_type  TEXT         NOT NULL CHECK (source_type IN ('file_upload', 'github_import', 'builtin')),
    source_name  TEXT         NOT NULL,            -- original filename, repo URL, or 'Vim Built-ins'
    github_url   TEXT,                             -- NULL unless source_type = 'github_import'
    parsed_at    TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    keymap_count INT          NOT NULL DEFAULT 0 CHECK (keymap_count >= 0),
    created_at   TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at   TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    deleted_at   TIMESTAMPTZ,                      -- soft delete

    -- github_url must be present for github imports, absent for others
    CONSTRAINT chk_github_url CHECK (
        (source_type = 'github_import' AND github_url IS NOT NULL)
        OR
        (source_type != 'github_import' AND github_url IS NULL)
    )
);

-- Primary access pattern: list all sources for a user (Sources page, keymaps list filter)
CREATE INDEX idx_keymap_sources_user_id
    ON keymap_sources(user_id)
    WHERE deleted_at IS NULL;

-- Duplicate detection: prevent re-importing the same file or repo
CREATE INDEX idx_keymap_sources_user_name
    ON keymap_sources(user_id, source_name)
    WHERE deleted_at IS NULL;
```

### 3. keymaps

The central practice table. Stores all keymap entries: user-imported from files/GitHub and built-in Vim motions. Built-in keymaps use `is_builtin = TRUE` and a system `user_id` (a reserved UUID constant defined in the application). The SRS system applies uniformly to both built-in and user-imported keymaps.

Soft delete allows users to remove keymaps from their active practice set without losing SRS history. The partial unique index on `(user_id, key_sequence, mode)` excludes soft-deleted rows so the same binding can be re-imported after removal.

```sql
CREATE TABLE keymaps (
    id             UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id        UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    source_id      UUID         REFERENCES keymap_sources(id) ON DELETE SET NULL,  -- NULL for builtins
    key_sequence   TEXT         NOT NULL,           -- raw key sequence: "<leader>ff", "gd", "ciw", "<C-p>"
    mode           TEXT         NOT NULL CHECK (mode IN ('n', 'i', 'v', 'x', 'o', 't', 'c')),
    description    TEXT         NOT NULL DEFAULT '',
    category       TEXT         NOT NULL DEFAULT 'other'
                                CHECK (category IN ('motion', 'leader', 'lsp', 'navigation', 'editing', 'plugin', 'other')),
    is_builtin     BOOLEAN      NOT NULL DEFAULT FALSE,
    created_at     TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at     TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    deleted_at     TIMESTAMPTZ                       -- soft delete
);

-- Uniqueness: one key_sequence+mode per user in the active set
-- Soft-deleted keymaps are excluded so re-import is allowed after removal
CREATE UNIQUE INDEX idx_keymaps_user_sequence_mode
    ON keymaps(user_id, key_sequence, mode)
    WHERE deleted_at IS NULL;

-- Practice session fetch: GetForPractice filters by user + mode
CREATE INDEX idx_keymaps_user_mode
    ON keymaps(user_id, mode)
    WHERE deleted_at IS NULL;

-- Analytics category breakdown and flashcard deck selection
CREATE INDEX idx_keymaps_user_category
    ON keymaps(user_id, category)
    WHERE deleted_at IS NULL;

-- Built-in Motion Trainer keymaps: shared lookup across all users
-- Used when user has no imported keymaps to populate a session
CREATE INDEX idx_keymaps_builtin
    ON keymaps(is_builtin, category)
    WHERE is_builtin = TRUE AND deleted_at IS NULL;

-- SRS daily queue: fetch keymap details for a list of IDs (ANY array operator)
-- Served by PK index; no additional index needed
```

### 4. practice_sessions

One row per practice session. Stores aggregate session metrics. Individual challenge data is in `practice_attempts`. Sessions can be in-progress (`completed_at IS NULL`), completed, or abandoned (never completed). Analytics queries always filter to `completed_at IS NOT NULL`.

The `mode` column matches the frontend session type selector: `practice` (user keymaps), `motion` (built-in motions), `leader` (leader key subset), `flashcard` (SRS-driven review).

```sql
CREATE TABLE practice_sessions (
    id                UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id           UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    mode              TEXT         NOT NULL CHECK (mode IN ('practice', 'motion', 'leader', 'flashcard')),
    started_at        TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    completed_at      TIMESTAMPTZ,                         -- NULL = in progress or abandoned
    total_attempts    INT          NOT NULL DEFAULT 0 CHECK (total_attempts >= 0),
    correct_attempts  INT          NOT NULL DEFAULT 0 CHECK (correct_attempts >= 0),
    accuracy          NUMERIC(5,2) CHECK (accuracy >= 0 AND accuracy <= 100),  -- percentage 0-100
    avg_response_ms   INT          CHECK (avg_response_ms >= 0),
    score             INT          NOT NULL DEFAULT 0 CHECK (score >= 0),
    streak_achieved   INT          NOT NULL DEFAULT 0 CHECK (streak_achieved >= 0),

    CONSTRAINT chk_correct_lte_total CHECK (correct_attempts <= total_attempts)
);

-- Analytics: date-range aggregations (accuracy trend, daily practice time)
-- Partial index excludes incomplete sessions which are never in analytics
CREATE INDEX idx_sessions_user_started
    ON practice_sessions(user_id, started_at DESC)
    WHERE completed_at IS NOT NULL;

-- In-progress session lookup: used to validate session ownership before recording attempts
CREATE INDEX idx_sessions_user_active
    ON practice_sessions(user_id, id)
    WHERE completed_at IS NULL;

-- Mode breakdown analytics
CREATE INDEX idx_sessions_user_mode
    ON practice_sessions(user_id, mode)
    WHERE completed_at IS NOT NULL;
```

### 5. practice_attempts

One row per answered challenge. Highest-volume table — grows at `sessions × session_length` rows per user. A dedicated user doing 1 session of 20 commands daily generates ~7,300 rows/year. At 1,000 active users: ~7.3M rows/year.

`user_input` stores exactly what the user typed, used for post-session review ("you typed X, correct answer was Y"). `response_ms` is the time from challenge display to user submission in milliseconds.

```sql
CREATE TABLE practice_attempts (
    id           UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id   UUID         NOT NULL REFERENCES practice_sessions(id) ON DELETE CASCADE,
    keymap_id    UUID         NOT NULL REFERENCES keymaps(id) ON DELETE CASCADE,
    user_input   TEXT         NOT NULL DEFAULT '',     -- what the user typed
    is_correct   BOOLEAN      NOT NULL,
    response_ms  INT          NOT NULL CHECK (response_ms >= 0),
    attempted_at TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);

-- Session drill-down: fetch all attempts in a session (session results screen)
CREATE INDEX idx_attempts_session
    ON practice_attempts(session_id, attempted_at ASC);

-- Analytics: per-user date-range accuracy aggregations (accuracy trend, most missed)
-- Composite covers: WHERE session_id IN (...) AND attempted_at BETWEEN ? AND ?
-- Joined to keymaps via keymap_id
CREATE INDEX idx_attempts_keymap_date
    ON practice_attempts(keymap_id, attempted_at DESC);

-- SRS weakest-keymap query: join to sessions to get user_id context
-- Used in: SELECT keymap_id, accuracy FROM practice_attempts pa
--          JOIN practice_sessions ps ON ps.id = pa.session_id
--          WHERE ps.user_id = ? AND pa.attempted_at > ?
CREATE INDEX idx_attempts_session_correct
    ON practice_attempts(session_id, keymap_id, is_correct);
```

### 6. spaced_repetition_records

SM-2 state per (user, keymap) pair. One row per keymap per user, created on first attempt via UPSERT. Updated after every attempt. `next_review_at` is computed by the SM-2 algorithm and determines when the keymap appears in the daily queue and flashcard sessions.

`ease_factor` floor is 1.3 (SM-2 standard). `interval_days` starts at 1, increases per correct answer weighted by `ease_factor`. `repetitions` counts consecutive correct answers (resets to 0 on incorrect).

```sql
CREATE TABLE spaced_repetition_records (
    id                UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id           UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    keymap_id         UUID         NOT NULL REFERENCES keymaps(id) ON DELETE CASCADE,
    interval_days     INT          NOT NULL DEFAULT 1 CHECK (interval_days >= 1),
    ease_factor       NUMERIC(4,2) NOT NULL DEFAULT 2.50
                                   CHECK (ease_factor >= 1.30 AND ease_factor <= 2.50),
    repetitions       INT          NOT NULL DEFAULT 0 CHECK (repetitions >= 0),
    next_review_at    TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    last_reviewed_at  TIMESTAMPTZ,                        -- NULL = never reviewed
    total_reviews     INT          NOT NULL DEFAULT 0 CHECK (total_reviews >= 0),
    correct_reviews   INT          NOT NULL DEFAULT 0 CHECK (correct_reviews >= 0),

    -- One SRS record per user+keymap pair
    CONSTRAINT uq_srs_user_keymap UNIQUE (user_id, keymap_id),
    CONSTRAINT chk_correct_lte_total CHECK (correct_reviews <= total_reviews)
);

-- Daily queue generation: keymaps due for review (overdue first)
CREATE INDEX idx_srs_user_due
    ON spaced_repetition_records(user_id, next_review_at ASC);

-- Weakest keymaps: sort by accuracy (correct_reviews / total_reviews) ascending
-- Used in daily queue composition and analytics
CREATE INDEX idx_srs_user_reviews
    ON spaced_repetition_records(user_id, total_reviews, correct_reviews)
    WHERE total_reviews > 0;

-- Never-practiced keymaps: keymaps user has never reviewed
CREATE INDEX idx_srs_user_new
    ON spaced_repetition_records(user_id)
    WHERE total_reviews = 0;

-- Flashcard due query: keymaps where next_review_at <= NOW()
-- Covered by idx_srs_user_due above
```

### 7. daily_queues

One row per user per UTC calendar date. Generated once on the day's first dashboard visit and fixed for the day. The `keymap_ids` array stores 20 ordered keymap UUIDs. `completed_at` is set when the user finishes all 20. UNIQUE constraint prevents duplicate queue generation if multiple requests race on the same day.

```sql
CREATE TABLE daily_queues (
    id            UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id       UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    queue_date    DATE         NOT NULL,                  -- UTC calendar date: '2026-06-16'
    keymap_ids    UUID[]       NOT NULL,                  -- ordered array of 20 keymap UUIDs
    generated_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    completed_at  TIMESTAMPTZ,                            -- NULL = queue not yet finished

    -- One queue per user per day (no duplicates even under concurrent requests)
    CONSTRAINT uq_daily_queue_user_date UNIQUE (user_id, queue_date)
);

-- Today's queue lookup: called on every dashboard load
-- Covers: WHERE user_id = ? AND queue_date = CURRENT_DATE
CREATE INDEX idx_daily_queues_user_date
    ON daily_queues(user_id, queue_date DESC);
```

### 8. achievements

Static definitions of all achievements. Seeded once at startup, never modified by application logic. 10 rows — no additional indexes needed beyond PK and the unique `slug` constraint. PostgreSQL will seq-scan 10 rows faster than using a B-tree index.

```sql
CREATE TABLE achievements (
    id               UUID  PRIMARY KEY DEFAULT gen_random_uuid(),
    slug             TEXT  NOT NULL UNIQUE,              -- machine code: 'first_session'
    name             TEXT  NOT NULL,                     -- display name: 'First Session'
    description      TEXT  NOT NULL,                     -- shown in achievement modal
    icon_name        TEXT  NOT NULL,                     -- Lucide icon name: 'trophy', 'flame'
    condition_type   TEXT  NOT NULL CHECK (
                         condition_type IN (
                             'session_count',    -- total completed sessions
                             'streak_days',      -- consecutive daily queue completions
                             'accuracy_pct',     -- accuracy in a single session
                             'mastered_count',   -- keymaps with >= 80% accuracy and >= 5 reviews
                             'category_mastery'  -- all keymaps in a category mastered
                         )
                     ),
    condition_value  INT   NOT NULL CHECK (condition_value > 0)  -- threshold for the condition_type
);
```

### 9. user_achievements

Junction table: which users have earned which achievements and when. Composite primary key eliminates a surrogate UUID and enforces uniqueness. `ON CONFLICT DO NOTHING` allows safe re-evaluation of achievement conditions without duplicates.

```sql
CREATE TABLE user_achievements (
    id             UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id        UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    achievement_id UUID         NOT NULL REFERENCES achievements(id) ON DELETE CASCADE,
    unlocked_at    TIMESTAMPTZ  NOT NULL DEFAULT NOW(),

    -- Prevent duplicate unlocks
    CONSTRAINT uq_user_achievement UNIQUE (user_id, achievement_id)
);

-- Profile page: all achievements earned by a user
CREATE INDEX idx_user_achievements_user
    ON user_achievements(user_id, unlocked_at DESC);

-- Recent achievements feed (home dashboard widget)
-- Covered by idx_user_achievements_user ordered by unlocked_at DESC
```

### 10. user_settings

One row per registered user. Created on first settings save (not at registration). Application defaults are applied in code when this row is absent. UPSERT pattern handles both first save and subsequent updates.

```sql
CREATE TABLE user_settings (
    id                      UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id                 UUID         NOT NULL UNIQUE REFERENCES users(id) ON DELETE CASCADE,
    theme                   TEXT         NOT NULL DEFAULT 'dark'
                                         CHECK (theme IN ('dark', 'light', 'system')),
    session_duration_minutes INT         NOT NULL DEFAULT 10
                                         CHECK (session_duration_minutes IN (5, 10, 15, 20, 30)),
    sounds_enabled          BOOLEAN      NOT NULL DEFAULT TRUE,
    animations_enabled      BOOLEAN      NOT NULL DEFAULT TRUE,
    keyboard_layout         TEXT         NOT NULL DEFAULT 'qwerty'
                                         CHECK (keyboard_layout IN ('qwerty', 'dvorak', 'colemak')),
    leader_key_symbol       TEXT         NOT NULL DEFAULT '<leader>'
                                         CHECK (
                                             leader_key_symbol IN ('<leader>', '<space>', '\', ',', 'g')
                                         ),
    created_at              TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at              TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);

-- All lookups are by user_id (unique constraint above acts as index)
-- No additional indexes required
```
